# Auto Garage Management System - Prisma Setup

## Overview

This project has been migrated from Supabase to **Prisma ORM** with a local PostgreSQL database.

## Tech Stack

- **Frontend**: React + TypeScript + Vite
- **Backend**: Express.js + TypeScript
- **Database**: PostgreSQL (via Prisma Postgres local server)
- **ORM**: Prisma 7
- **Auth**: JWT with httpOnly cookies
- **File Upload**: Multer (files stored in `public/uploads/garages`)

## Getting Started

### 1. Install Dependencies

```bash
npm install
```

### 2. Start Prisma Postgres Server

The local PostgreSQL server is automatically managed by Prisma:

```bash
npx prisma dev
```

This runs in the background and creates the `DATABASE_URL` in `.env`.

### 3. Run Database Migrations

```bash
npm run prisma:migrate
```

### 4. Start Development Servers

**Option A: Start both servers together**
```bash
npm run dev:all
```

**Option B: Start separately**
```bash
# Terminal 1 - Backend API (port 3001)
npm run dev:server

# Terminal 2 - Frontend (port 5176)
npm run dev
```

### 5. Access the Application

- **Frontend**: http://localhost:5176
- **API**: http://localhost:3001/api
- **Prisma Studio** (Database GUI): `npm run prisma:studio`

## API Endpoints

### Authentication
- `POST /api/auth/register` - Create new user
- `POST /api/auth/login` - Login user
- `POST /api/auth/logout` - Logout user
- `GET /api/auth/me` - Get current user

### Garages
- `GET /api/garages` - List all garages
- `GET /api/garages/:id` - Get single garage
- `POST /api/garages` - Create garage (authenticated, multipart/form-data)
- `PUT /api/garages/:id` - Update garage (authenticated, multipart/form-data)
- `DELETE /api/garages/:id` - Delete garage (authenticated)

## Features

### Implemented
✅ User authentication (register/login/logout)
✅ Create garage with file upload
✅ List all garages
✅ View garage details
✅ Owner-only edit/delete permissions
✅ File upload for logos
✅ PostgreSQL database with Prisma

### Authentication Flow
1. User registers/logs in
2. JWT token stored in httpOnly cookie
3. Frontend sends cookie with each request
4. Backend verifies token in middleware
5. Protected routes check user ownership

### File Upload Flow
1. User selects logo file in AddGarage form
2. Frontend shows image preview
3. On submit, file sent as multipart/form-data
4. Backend validates file type (images only, max 5MB)
5. File saved to `public/uploads/garages/`
6. File path `/uploads/garages/<filename>` stored in database
7. Frontend displays uploaded image

## Troubleshooting

### Backend won't start
- Ensure Prisma Postgres server is running: `npx prisma dev`
- Check DATABASE_URL in `.env`
- Regenerate Prisma client: `npm run prisma:generate`

### Database connection errors
- Restart Prisma server: `npx prisma dev`
- Check if port 51213 is available
- Run migrations: `npm run prisma:migrate`

### File uploads fail
- Ensure `public/uploads/garages/` directory exists
- Check file size (max 5MB)
- Verify file type (jpeg, jpg, png, gif, webp only)

## Production Deployment

1. **Database**: Use hosted PostgreSQL (e.g., Neon, Supabase, Railway)
2. **Update DATABASE_URL** in production `.env`
3. **Run migrations**: `npx prisma migrate deploy`
4. **File Storage**: Use cloud storage (S3, Cloudinary) instead of local uploads
5. **Environment**: Set `NODE_ENV=production`
6. **JWT_SECRET**: Use strong random secret
7. **CORS**: Update `FRONTEND_URL` to production domain

## Notes

- Prisma 7 requires `@prisma/adapter-pg` for `prisma+postgres://` URLs
- File uploads stored locally - consider cloud storage for production
- JWT tokens valid for 7 days
- Database automatically created by `prisma migrate dev`
